# v0.1.0 Release — First stable
**Status: DRAFT — awaiting approval**
**Date: 2026-06-14**

## Goal

Promote the v0.0.10 release codebase to the first official stable release (`v0.1.0`). No new features, code changes, or refactoring will be performed in this release cycle. The scope is limited strictly to updating release version properties, configuring the final production update URLs, establishing signed APK generation pipelines, and updating repository release notes.

## Scope (in)

1. **Version Code Bump**:
   - Increment `versionName` to `"0.1.0"` and increment `versionCode` inside the gradle configuration.

2. **Production Update Check Configuration**:
   - Swap the update channel URL placeholder with the real GitHub Releases JSON endpoints (e.g., using production resources inside `strings.xml` or custom build configs).

3. **Keystore Signing Configuration**:
   - Update `signingConfigs` inside the gradle application module to conditionally load keystore metadata from a `keystore.properties` file located at the project root.
   - User must supply their own keystore files and matching credentials properties. The agent will ask and stop if these assets are absent, rather than generating mock keystores.

4. **Release Notes Generation**:
   - Create a structured release notes document aggregating all changelog entries from `v0.0.1` through `v0.0.10`.

5. **AI Documentation Release Gate**:
   - Perform documentation sweeps checking that all files in `AI/` are updated.
   - Assert that no unresolved `Severity-HIGH` issues remain open inside [KNOWN_ISSUES.md](file:///d:/Repos/splitandmerge/AI/KNOWN_ISSUES.md).

6. **Drafting GitHub Release Information**:
   - Prepare copy-paste text for the GitHub Release containing APK file links, SHA-256 validation checksums, exact file sizing info, and installation steps.

7. **Artwork Integration (Optional)**:
   - If the user provides official logo designs or icons, substitute the vector-based adaptive launcher icon placeholders created in `v0.0.9` with these custom visual assets.

## Out of scope

- All code refactors, screen edits, or layout overrides.
- Changes to engine operations or dependencies.
- Automated pipeline deployment execution.

## Acceptance criteria (measurable)

1. **Successful Signed Build**: The Gradle `assembleRelease` task compiles successfully, outputting a signed production APK file.
2. **Keystore Integrity**: The output APK signature checks out correctly against the user's release certificate.
3. **Correct Version Metadata**: The built APK reports `versionName = "0.1.0"` and the bumped `versionCode`.
4. **Working Update Connection**: Connecting to the production version manifest reports that the app is fully up-to-date.
5. **No Regression**: Round-trip splits and merge operations continue to execute successfully.

## Rollback plan

If the stable release has issues, we can check out the tag prior to version updates (`v0.0.10`) and revert gradle parameters back to the debug configuration.
