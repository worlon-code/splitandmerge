# v0.0.11 Release — Transport (Byte-Exact) Split Mode & Merge Auto-Detection

Introduce a byte-exact "Transport" split mode that splits files using pure Java byte chopping without FFmpeg, yielding byte-identical files upon merging (guaranteeing exact SHA-256 match). Merge operations will auto-detect whether parts are structural or transport-split, and route execution accordingly.

## Goal

Provide a fast, byte-exact splitting alternative ("Transport" split) that does not remux or decode the video, producing `.bin.NNN` slices. This is intended for cloud transport hosts (such as rclone, Telegram, Backblaze, etc.) that do not validate video formats. At merge time, the application automatically Sniffs the input parts (via manifest, file extensions, or magic bytes) and joins them via pure Java stream copy, ensuring 100% byte-identical reconstruction of the original file.

## Scope (in)

1. **Domain Model & Enum Extensions**:
   - Extend/introduce a mechanism for splitting mode distinction. As specified, the `SplitMode` enum is extended with `STRUCTURAL` (default, existing remuxing behavior) and `TRANSPORT` (new byte-chop behavior).
   - Backward compatibility: any persisted job without a specified split type/mode will default to `STRUCTURAL` on read.
   - Core Transport Engine components:
     - `TransportSplitter.kt` (pure Java I/O splitting).
     - `TransportMerger.kt` (pure Java I/O merging).
     - `PartModeDetector.kt` (determines if selected parts are structural or transport-split).
   - Sidecar manifest `SplitManifest.kt`:
     - `@Serializable` data class written next to transport parts as `<base>.split.json`.
     - Stores metadata: tool version, split type, source size, source SHA-256, and part information.
     - Optional at merge time; if present, used to verify integrity by comparing output SHA-256 with `sourceSha256`.

2. **Split UI (Settings & Per-Job Toggle)**:
   - **SettingsScreen**:
     - Add a "Default split mode" setting row with Radio buttons: `Structural (default)` and `Transport (byte-exact)`.
     - Stored in `SettingsRepository` preference flows (defaults to `STRUCTURAL`).
   - **SplitConfigScreen**:
     - Add a Switch toggle near the capacity slider: `"Transport (byte-exact) split"`.
     - Switch defaults to the preference value in Settings.
     - When switched ON:
       - Show inline warning text below the switch: *"Parts are not individually playable. Use this only for cloud transports that don't validate video (rclone, Telegram, 7z-style). Do NOT use for Google Photos, iCloud, or anywhere you might want to play one part on its own."*
       - Hide ByParts UI options (Transport mode always splits by size).

3. **TransportSplitter Execution**:
   - Inputs: Source `Uri`, part size limit `partSizeBytes` (Long), output directory `Uri`, and `outputBaseName`.
   - Process:
     - Open source file via `RandomAccessFile` (using real path if resolved, or SAF descriptor otherwise).
     - Allocate an 8 MB `ByteBuffer` chunk buffer.
     - Read and write sequentially to output parts named `<base>.bin.001`, `<base>.bin.002`, ..., `<base>.bin.NNN`. The `.bin` extension acts as a clear indicator that the file is not individually playable.
     - Calculate SHA-256 hashes of the source file and each individual part in a single-pass streaming fashion.
     - Upon completion, write `<base>.split.json` containing the metadata manifest.
   - Progress Reporting: Emits % completion and EMA byte transfer speed (MB/s) matching the existing `Splitter` progress interface, requiring no changes to `JobProgressScreen`.

4. **PartModeDetector (Auto-Detection)**:
   - Evaluates a list of selected part `Uri`s in order of priority:
     - **a) Manifest present**: If `<base>.split.json` is selected, parse and trust `manifest.type`.
     - **b) Naming convention (Transport)**: If all filenames end with `.bin.NNN`, return `TRANSPORT`.
     - **c) Naming convention (Structural)**: If all filenames match the `*.partNNN.*` format, return `STRUCTURAL`.
     - **d) Magic-Byte Sniff**: Open each file and read the first 4 bytes. Compare against EBML magic bytes (`0x1A 0x45 0xDF 0xA3`):
       - All parts contain EBML magic → `STRUCTURAL`.
       - Only the first part contains EBML magic, and others do not → `TRANSPORT`.
       - Mixed EBML magic positions → Reject as `MIXED`.
       - No EBML magic in any part → Reject as `INVALID`.
   - Optimized execution: Probe costs only 4 bytes read per descriptor open.

5. **Merge UI Integration**:
   - **MergeOrderScreen**:
     - Run `PartModeDetector` on selected URIs when loading the screen.
     - Render an info chip near the file list: `"Structural"` (green container/text) or `"Transport (byte-exact)"` (blue container/text).
     - If detector returns `MIXED` or `INVALID`, display an `AlertDialog` blocking progress: *"These files mix structural and byte-split parts (or aren't valid video). Pick parts from a single split."*
     - If `TRANSPORT` mode is active, and no manifest or `.bin.NNN` naming conventions are found (e.g. plain `.mkv` files only):
       - Show a banner above the list: *"Byte-split parts detected. The first part (the one that starts with a video header) goes first. The remaining parts must be in original split order."*
       - Detector automatically places the EBML-magic-carrying file at index 0 by default.

6. **TransportMerger Execution**:
   - Inputs: Ordered `List<Uri>` of parts, optional manifest `Uri`, and output `Uri`.
   - Process:
     - Open SAF output stream.
     - For each part URI, open an input stream and copy chunks using an 8 MB buffer.
     - Dynamically calculate the output file's SHA-256 hash as bytes are written.
     - Verify output:
       - If manifest is present: verify that output SHA-256 matches `manifest.sourceSha256`. If it does not, delete output and throw a verification exception: *"Merged file does not match original (corrupt, missing, or wrong order)."*
       - If no manifest: complete write successfully, and show a note on `MergeResultScreen`: *"No manifest available. Verify playback before deleting source parts."*
   - Progress Reporting: Emits % and MB/s progress matching standard Merger events.

7. **Dispatcher & Compatibility**:
   - Create a thin `MergerDispatcher` routing to either `Merger` (FFmpeg concat) or `TransportMerger` based on `PartModeDetector` result.
   - Existing FFmpeg `concat` merge path is unchanged.

---

## Out of scope

- Multi-volume compressed archive formats (`.7z`, `.rar`, etc.) splitting or extraction.
- Encryption, password protection, or obfuscation of byte slices.
- Cross-mode translation (e.g., trying to convert transport bin parts into structural streams during merge).
- Renaming user-provided source files to match standard `.bin.NNN` extensions.
- Automated cloud hosting API integration (Dropbox/Drive/Telegram upload/download pipelines).

## Acceptance criteria (measurable)

- **Default Behavior**: Default configuration remains `STRUCTURAL`. v0.0.10 structural split remains at baseline speeds ± 1s.
- **Byte Accuracy**: Merged output of a 25 GB MKV file under Transport mode must yield a byte-identical hash compared to the source file (SHA-256 verified).
- **Graceful Detection & Rejection**: Mixed structural and byte-split selections must be detected and blocked via an `AlertDialog` prompt, preventing process crashes.
- **Performance**: Pure byte-exact splitting must bypass FFmpeg remuxing overhead. A 9 GB transport split must execute in `< 0.5x` the time of a structural split on the same storage medium.
- **Magic-Byte Sniffing**: Transport splits packaged as plain `.mkv` files (with no manifest or suffix hints) must be correctly identified and the file containing the Ebml header sorted to index 0.

## Open questions

1. **SplitMode Enum Name Collision**:
   > [!IMPORTANT]
   > The existing enum `com.splitandmerge.mkvslice.domain.model.SplitMode` contains configuration values: `EXACT_PARTS`, `SIZE_CAP_ONLY`, and `BOTH`. These determine the structural calculation logic.
   >
   > Extending it with `STRUCTURAL` and `TRANSPORT` represents a collision of concerns (mixing split calculation strategies with split engine formats).
   >
   > **Proposed Resolution**: We must rename/refactor the existing enum to `SplitStrategy` (representing structural planner options) and define a new `SplitMode` enum containing `STRUCTURAL` and `TRANSPORT`. This will be detailed in the implementation phase.
   
2. **Magic Sniff False Positives**:
   - sniffs the first 4 bytes for Ebml header (`0x1A 0x45 0xDF 0xA3`). While standard, if a non-Matroska video format (like MP4/TS) is transport-split, it will not carry this header. Sniffing will fall back to `INVALID`. We should document that Transport auto-detection on headers is explicitly restricted to Matroska (`.mkv`) containers.
