# v0.0.7 Release — Tight Boundaries, Library Details, Merge Disk Optimization

**Status: APPROVED-WITH-AMENDMENTS**
**Date: 2026-06-14**

---

## Problem

1. **Fix 1 (Tight split boundaries)**: Splitter overlaps adjacent parts by ~1 GOP because of timing snapping discrepancies or incorrect argument ordering. This inflates duration and creates a drift of ~3 seconds in split-then-merge round-trips.
2. **Fix 2 (Library details)**: Rows in the Library screen (S2) only navigate for `RUNNING` (S7 Progress) and `DONE` (S8/S13 Result) jobs. `CANCELLED` and `FAILED` jobs are non-tappable, so the user cannot see why a job failed or retry it.
3. **Fix 3 (Merge disk optimization)**: Staging all parts and copy-out at the end uses up to 2× input size in the cache folder. For a 24 GB merge, it consumes 52 GB peak disk and takes ~7 minutes of redundant I/O.

---

## Proposed change

### 1. Fix 1: Tight Split Boundaries (DEFERRED-TO-v0.0.8)
*Note: v0.0.6 +3s drift is at the source keyframe granularity floor. Further tightening requires deeper investigation of -copyts + -map 0 audio EOF behaviour. Not blocking v0.0.7 release.*

### 2. Fix 2: Library Row Navigation & Job Detail Sheet
- Introduce a sealed MVI intent interface `LibraryIntent` and ViewModel routing `NavCommand` in `LibraryViewModel`.
- Tap on `CANCELLED` or `FAILED` row opens a new `JobDetailSheet` bottom sheet.
- `JobDetailSheet` is a Material 3 `ModalBottomSheet` hosted in `LibraryScreen` showing:
  - Status icon, chip, and title.
  - Relative time or formatted time (e.g. "Cancelled at 14:23").
  - For `FAILED`: error message in a tonal container + "Show details" expandable tail.
  - Actions:
    - "Retry": Spawns a new queued job with a fresh UUID. `PartEntity` rows for MERGE retries must be inserted with NEW row IDs, NEW `jobId` pointing to the new `JobEntity`, and the SAME SAF URIs verbatim. The original FAILED row and its `PartEntities` remain unchanged.
    - "Delete row": Deletes the row.
  - On tablet: cap the bottom sheet's max width at `600dp` on screen widths `>= 800dp` to avoid stretching across the full screen.

### 3. Fix 3: Disk Optimization & Pre-flight
- **Path Resolution**: Create `MergePathResolver` to resolve SAF tree/document URIs and content URIs to real paths (e.g. `/storage/emulated/0/...`).
- **Storage Pre-flight**: Update `StoragePreflight` to verify cache space depending on resolved paths:
  - All real: 0 bytes (needs only free space on output volume >= `sum(parts) * 1.05`).
  - Inputs staged: `max(part size)`.
  - Output not real: `sum(part sizes)`.
  - Both staged: `max(part size) + sum(part sizes)`.
- **Sequential Cleanup**: During `CONCAT` phase in `Merger.kt`, when progress `timeMs` passes the end boundary of part N plus a 5-second safety margin, delete its local staged file immediately. The concat demuxer does NOT seek backwards; the real risk is brief overlap during child transition (typically 1-2 s), so we use a safety guard:
  ```kotlin
  val cleanupGuardSeconds = 5.0
  val safeBoundary = partEndTimesSeconds[deleteIndex] + cleanupGuardSeconds
  if (timeMs / 1000.0 < safeBoundary) break
  stagedParts[deleteIndex].delete()
  deleteIndex++
  ```
  - Never delete the last staged part during `CONCAT` progression; delete it only in the `finally{}` block, after the engine flow completes.
- **Phase Label Updates**: Show `Step X of Y` where Y is the total active phases (1, 2, or 3).
  - When steps == 1 (all-real fast path) or when the inputs-staged-only path has only a single step/phase (or steps <= 1), drop the "Step X of Y" prefix and just show "Merging".
  - For example, if only a single phase is executed (steps == 1), display "Merging" rather than "Step 1 of 1 · Merging".
