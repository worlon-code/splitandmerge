# v0.0.10 — Work Summary

## Goal
Add an opt-in disk fast-path for the merge pipeline to reduce cache pressure on devices with usable direct file access. Keep the proven stage-then-concat pipeline as the default.

## Scope decisions
- Originally scoped Fix 1 (audio-snap) + Fix 3 (fast-path).
- Narrowed to Fix 3 only after baseline drift was deemed acceptable and audio-snap design needed more thought.
- K-018 created to track Fix 1 for v0.0.12.

## What shipped
- Merger.kt: SettingsRepository injection; canFastPath gate with input-readability + output-writability probes; effectiveCanFastPath replaces canFastPath in all 5 downstream branches; graceful staging fallback throughout.
- MergerFastPathTest.kt: 4 unit tests covering the gate.
- Ripple: MergerArgvTest + MergerCollisionTest updated to pass SettingsRepository to the new Merger constructor.

## What was tried and abandoned
- Q2 first proposal rewrote MergePathResolver to delegate to PathResolver. Rejected as scope creep; existing resolver handles all 4 URI shapes we need.
- First Q1 proposal threw IllegalStateException on output resolution failure. Replaced with effectiveCanFastPath pattern + graceful fallback.

## Issues caught in review
- Output path mismatch (tree URI vs document URI). Fixed with explicit candidate construction from resolved tree dir + dir name + file name.
- newFile cleanup on STAGED failure. Pre-existing flag plumbed through to the new earlier createFile site.
- Hidden test file edits revealed after initial Step 2 submission. Disclosure rules tightened.
- First Step 2 batch used Timber tree planting and system-tmp shared paths. Cleaned up to TemporaryFolder rule, no log noise, mode-agnostic verify() calls.

## What failed on-device, and why
- Step 3 v1: fast-path crashed with JsonDecodingException. Root cause: scoped-storage blocks native read of SAF paths. Diagnosed via run-as cat. Fixed via Step 3.5 probes.
- Step 3-v2: fast-path probes correctly detected the block and fell back to staging. Confirmed via Timber warning in Phase 5 logcat.

## Verification summary
- Device: Samsung M51 (RZ8N90X627R).
- Source: Perfect Crown S01E04 (3-part split).
- Container size match: 1,028,760,434 bytes both runs.
- Post-4KB SHA-256 match: confirms identical A/V streams.
- 75 byte diffs in header region (SegmentUID / WritingApp).
- No crashes, no Permission denied, no SAF errors visible to user.

## Caveats
- Fast-path is effectively a no-op on the realistic distribution (targetSdk=35, SAF-only). K-019 commits to a FileSystem seam in v0.0.11; we will then decide whether fast-path stays as conditional code or is removed.

## SECTION F — TERMS & EXAMPLES

### 1. SHA-256
* **What it is**: A cryptographic hash function that generates a unique 64-character hexadecimal signature representing the exact contents of a file. If even a single byte of the file changes, the signature changes completely.
* **Example from this run**: The raw signature `D327C457DFC8EBC61A698F009EE27E55D963F8D87D98AF2D54AF09EA8BA58E01` computed for the `staged_output.mkv` file.
* **Why it mattered for v0.0.10**: It is used to verify that the fast-path merge output and standard merge output files are identical.

### 2. Container vs Stream
* **What it is**: A stream contains the raw, compressed audio or video data packets, whereas a container is the "wrapper" file format (like MKV or MP4) that packs these streams together along with synchronization metadata, subtitles, and header tags.
* **Example from this run**: The `1,028,760,434-byte` Matroska container wrapping approximately `1 GB` of combined H.264 video, AAC audio, and subtitle streams.
* **Why it mattered for v0.0.10**: We needed to verify that the raw video and audio streams matched exactly, even if the surrounding container headers had minor differences.

### 3. MKV / Matroska
* **What it is**: An open-standard, highly flexible container format capable of holding an unlimited number of video, audio, picture, or subtitle tracks in one file.
* **Example from this run**: The output files `staged_output.mkv` and `fastpath_output.mkv` generated by our merge engine.
* **Why it mattered for v0.0.10**: It is the primary target format merged by the app, and its unique binary structure dictates how we compare outputs.

### 4. EBML
* **What it is**: Extensible Binary Meta Language; a binary XML-like markup format used as the foundation for Matroska (MKV) files. It begins with a specific "magic number" to identify the file format.
* **Example from this run**: Both merged files start with the hex sequence `1A 45 DF A3` (the EBML magic number).
* **Why it mattered for v0.0.10**: Understanding EBML allowed us to parse the file structure and isolate the metadata header from the stream payload.

### 5. SegmentUID
* **What it is**: A random 128-bit unique identifier generated by the muxer and written to the Matroska Segment Information header block.
* **Example from this run**: The bytes at offset `220` where the two merges differed (`EA 88 07 05` in staged vs `CC EA DF F9` in fastpath) due to the SegmentUID being randomly regenerated on each run.
* **Why it mattered for v0.0.10**: It explains why the raw file SHA-256 hashes differed between two identical runs of the merger.

### 6. WritingApp / WritingDate
* **What it is**: Metadata tags stamped inside the Matroska header indicating which software library created the file and the exact timestamp when it was muxed.
* **Example from this run**: The `Lavf62.3.100` string (FFmpeg's libavformat library tag) and differing timestamp blocks written near offset `299` in our outputs.
* **Why it mattered for v0.0.10**: These variable values contribute to container-level SHA-256 mismatches despite identical media streams.

### 7. Mux / Muxer / Muxing
* **What it is**: The process of multiplexing, or combining, multiple separate media streams (audio, video, subs) into a single container file.
* **Example from this run**: FFmpeg's Matroska muxer assembling the split parts into a single `.mkv` output file.
* **Why it mattered for v0.0.10**: The muxer's generation of random segment IDs and timestamps determines the final binary structure of the merged file.

### 8. -c copy (lossless concat)
* **What it is**: An FFmpeg command flag that instructs the engine to copy the input audio and video packets directly to the output without decoding or re-encoding them, preserving original quality and running extremely fast.
* **Example from this run**: The engine arguments `-c copy` passed to FFmpeg to concatenate the parts losslessly.
* **Why it mattered for v0.0.10**: It ensures that the underlying audio/video stream packets remain completely identical, allowing us to use a post-header hash check for validation.

### 9. Cluster
* **What it is**: The Ebml sub-element in MKV files that wraps the actual timestamped media packets (video frames, audio blocks).
* **Example from this run**: The `1,028,756,338 bytes` of data following the first 4 KB header containing identical cluster blocks.
* **Why it mattered for v0.0.10**: It constitutes more than 99.99% of the file, proving that the media streams did not change.

### 10. Fast-path vs Staged path
* **What it is**: The staged path copies user files to the app's internal cache before running FFmpeg, consuming extra disk space. The fast-path bypasses this copy, feeding input paths directly to FFmpeg to save space and time.
* **Example from this run**: In Phase 5, the fast-path probes failed and fell back to the staged path at `21:17:40.714` as a safeguard.
* **Why it mattered for v0.0.10**: Implementing the fast-path, while safely falling back to staging on scoped storage, is the core objective of the v0.0.10 update.

### 11. canFastPath / effectiveCanFastPath / improveReliability
* **What it is**: Settings flag (improveReliability) + two computed gates inside Merger.
* **Example from this run**: 
  - `improveReliability = false` → fast-path considered
  - readability probe fails → `canFastPath = false`
  - `canFastPath = false` → `effectiveCanFastPath = false`
  - Merger executes staged path (same as if `improveReliability` had been true)
* **Why it mattered for v0.0.10**: These gates ensure the fast-path optimization only engages when it is safe to do so.

### 12. Probe (readability / writability)
* **What it is**: Lightweight filesystem operations (a 1-byte read test on inputs and a temporary file creation on the output directory) used to check if native code can access the files.
* **Example from this run**: The `FileInputStream.read()` check on the resolved parts, which failed for `/storage/emulated/0/Download/.../part001.mkv`.
* **Why it mattered for v0.0.10**: These probes prevent permission failures by checking read/write permissions before executing FFmpeg commands.

### 13. SAF (Storage Access Framework)
* **What it is**: Android's system framework for picking and accessing files. It grants temporary content URIs to the app, but these virtual paths cannot be opened directly by native C/C++ binaries.
* **Example from this run**: The `content://outputDir` URI representing the target directory selected via the SAF picker.
* **Why it mattered for v0.0.10**: Native binaries (FFmpeg) spawned by the app do not inherit SAF permissions, making direct file path access fail on scoped storage.

### 14. targetSdk
* **What it is**: The Android API level the app is compiled against, which determines which system behavior changes and security constraints are enforced.
* **Example from this run**: `targetSdk = 35` (Android 15), which enforces strict scoped storage.
* **Why it mattered for v0.0.10**: Enforcing strict scoped storage prevents the app from accessing raw storage paths, making the probes fail and trigger the staged fallback.

### 15. MediaProvider
* **What it is**: The Android system component that manages storage access. It intercepts standard filesystem API calls and denies them if the app lacks explicit permissions.
* **Example from this run**: The logcat error `E/MediaProvider: Permission to access file: ... is denied`.
* **Why it mattered for v0.0.10**: It is the system service that blocks FFmpeg's access to the file paths.

### 16. run-as
* **What it is**: An ADB shell utility that lets you run shell commands as if you were the specified app package, matching its exact UID and permission context.
* **Example from this run**: `run-as com.splitandmerge.mkvslice.debug` used to execute file access checks.
* **Why it mattered for v0.0.10**: It allowed us to simulate the app's permission context during debugging.

### 17. df vs du
* **What it is**: `df` (Disk Free) measures usage of the entire disk partition, while `du` (Disk Usage) measures the size of a specific directory.
* **Example from this run**: The partition-wide measurements we sampled, which were affected by background processes.
* **Why it mattered for v0.0.10**: Using `df` introduces noise from other apps, meaning we should use `du` on the app's cache directory in future tests.

### 18. Logcat
* **What it is**: Android's system-wide logging service, which records debug statements, warnings, and error messages from all running apps and system components.
* **Example from this run**: The warning captured in `phase5_logcat.txt` indicating a fallback to staging.
* **Why it mattered for v0.0.10**: It provided the logs verifying that the fallback logic executed as intended.
